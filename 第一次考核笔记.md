## 关于Github

1.Pull request与Fork等专业词语：

A：Pull request：它是我在自己的 Fork 副本中完成修改后，向本人仓库发起的 “请求将我的修改合并到原仓库” 的申请。

B：Fork:  将他人的 GitHub 仓库完整复制到自己的 GitHub 账号下，形成独立副本。

and so on



2.Github的学习笔记

接触 GitHub 两天，勉强摸清了基础用法。刚开始看到 “仓库、分支、Fork” 这些词时一头雾水，现在慢慢有了头绪。

首先，GitHub 就是代码的 云仓库，能存放项目代码还能跟踪修改，适合我们新手备份代码了。目前学会了创建仓库，给项目命名、写描述，知道了要加 README 文件说明项目用途。

然后是核心的协作工具，Fork 能把别人的仓库复制到自己账号下，随便修改不影响原项目；Pull Request 是向原作者提交修改建议的方式，不过目前还未实际操作过，只看了流程演示，后续会加紧安排。

最实用的是版本控制，每次修改代码后，用 commit 提交并写清楚修改内容，以后想回退到之前的版本都能找到。

目前还在熟悉基本操作，很多功能比如分支管理、冲突解决还不太懂，接下来打算试着 Fork 简单项目，跟着教程做一次完整的 PR，慢慢积累实战经验。





##  关于Markdown

1.学习笔记

刚听说 Markdown 时，以为又是复杂的工具类似vs code dev等 ，学了一点才发现是便捷的语言工具。不用点鼠标找格式按钮，用简单符号就能搞定排版，适合我们新手记笔记、写文档了。

目前先掌握了基础用法：# 加空格是标题，# 越多级别越低，最多到六级；星号或减号能做无序列表，数字加句号就是有序列表，整理思路超清晰。还学会了用 ** 加粗文字、* 斜体，引用别人的话就加 > 符号，看起来很规整。

最实用的是插入链接和图片，链接用 [显示文本](https://www.doubao.com/chat/链接地址)，图片只要在前面加个！就行。表格也没想象中难，用 | 分隔列、- 分隔表头，简单几步就能做出整齐的表格。

现在写笔记都用 Markdown，比 Word 省事儿多了。不过还有代码块、公式这些进阶功能没学会，接下来打算试着用它写项目 README 文件，把学到的用法多练几遍，争取熟练掌握所有基础功能。





## 关于本文案的规范性

在写本文案前我去了解了一些关于markdown的写作规范等，为其之简洁与便利所感慨，因此本文也按照部分md的格式来写。

1. **结构逻辑清晰**：以标题分级搭建框架，从基础的标题、文本格式，到进阶的列表、链接、代码块，按 “易→难” 顺序排列，模块划分明确，我能快速定位所需内容。
2. **表达直击重点**：每个模块仅保留核心知识点，代码块标注语言类型并附简洁示例，不额外添加冗余解释，避免信息干扰。
3. **格式辅助阅读**：用加粗突出模块标题，符合 Markdown  特性，又通过视觉区分让重点信息更醒目，整体做到 “内容不冗余、逻辑不混乱、重点不模糊”。





## 关于Markdown笔记的延申

## 一、文本与列表

1. **上下标**：`X^2^`（X²）、`H~2~O`（H₂O），需 Typora 等编辑器支持。
2. **高亮**：`==重点内容==`（== 重点内容 ==），醒目划考点。
3. **任务列表**：`- [x] 已完成`、`- [ ] 待办`，适合做计划。

## 二、公式与脚注

1. **公式**：行内用 `$E=mc^2$`（*E*=*m**c*2），块级用 `$$` 包裹复杂公式。
2. **脚注**：正文写 `[^1]`，文末注 `[^1]: 补充内容`，方便标注来源。

## 三、实用功能

1. **自动目录**：开头加 `[TOC]`，同步标题生成导航。
2. **代码块优化**：指定语言（````java`）实现高亮，用 `<details>` 标签实现代码折叠。
3. **注释**：`<!-- 仅编辑器可见 -->`，记录临时想法。



## 四、优秀模板与学习笔记

选择了「极致简洁型」和「功能丰富型」两个代表性案例，覆盖不同展示需求。

#### 案例 1：Abhishek Naidu（简洁高效型）

- **参考链接**：https://github.com/abhisheknaiidu/abhisheknaiidu
- 核心亮点
  1. 顶部用大标题明确身份，下方用图标 + 短句快速展示技术栈，无多余信息。
  2. 嵌入「访客计数器」和「GitHub 统计卡片」。
  3. 仅用黑白灰 + 少量蓝色点缀，重点信息用粗体突出。

#### 案例 2：Maximilian Stoiber（功能丰富型）

- **参考链接**：https://github.com/mxstbr/mxstbr
- 核心亮点
  1. 用 GitHub Actions 自动更新「近期动态」。
  2. 自定义徽章和进度条，增强记忆点。



#### 1. 内容设计技巧

- **身份前置**：开头必须明确「你是谁」「专注什么领域」，避免访客困惑。
- **数据可视化**：用 GitHub 统计卡片（如 github-readme-stats）、贡献图等替代文字描述，更直观。
- **动态更新**：借助 GitHub Actions 自动同步博客文章、项目进度，减少手动维护成本。

#### 2. 视觉优化技巧

- **色彩克制**：主色调不超过 3 种，优先选择黑白灰 + 1 个强调色（如蓝色、绿色），避免杂乱。
- **图标辅助**：用 Font Awesome 图标（如 🐳 代表 Docker）替代文字，缩短信息长度，提升美观度。
- **信息分区**：用分割线（---）或卡片式布局划分模块（技术栈、项目、联系方式），逻辑更清晰。





## 关于学习Github与Markdown遇到的困难与相应解决方案

刚接触 Github 时，我直接懵了 —— 仓库、分支、Commit 这些术语像天书，点进界面不知道该点哪里，更怕误操作删了东西。后来我跟着 B 站的新手教程，先建了个测试仓库，每步都暂停记笔记，终于搞懂 “新建仓库→上传文件→提交修改” 的基本流程。

学 Markdown 更头疼，记不住语法。想加粗文字却总输错符号，表格排版更是一团乱，预览时格式全错。我索性把常用语法（比如 #标题、*斜体*）抄在备忘录里，每次写东西就对照着用，还找了个在线编辑器边写边预览，练了 3 次就熟练多了。

原来新手不用怕出错，多试多查，再复杂的工具也能慢慢上手。
